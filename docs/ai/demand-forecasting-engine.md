# Demand Forecasting Engine

## Implemented State

Feature 3 is implemented as a backend statistical forecasting engine.

- Endpoint: `GET /api/admin/ai/demand-forecast`
- Backend module: `apps/api/src/modules/demand-forecasting`
- Shared contract: `packages/shared-types/src/demand-forecasting.ts`
- Audit model: `DemandForecastLog`
- Frontend component: `apps/web/src/components/admin/ai/DemandForecastPanel.tsx`
- Test script: `npm.cmd run test:demand-forecast --workspace @smart-restaurant/api`

This MVP does not use FastAPI, hosted LLMs, or external AI providers. Forecasts are computed inside the NestJS API from historical order data.

## Purpose

The engine predicts expected item demand for a branch on a selected future date. It helps admins and managers prepare inventory, kitchen staffing, and operational expectations before service.

## Where It Appears

- Admin analytics page: forecast summary cards and item-level forecast table.
- Admin dashboard: future cards can reuse the same endpoint for expected orders and revenue.
- KDS/admin insights: future views can derive peak hour and top expected items from the response.
- Inventory page: future inventory conversion can map expected menu item demand to ingredients.

## Endpoint

```txt
GET /api/admin/ai/demand-forecast
```

Query parameters:

```txt
branchId=seed-branch-1
date=2026-05-05
categoryId=optional-category-id
lookbackDays=30
```

Rules:

- `branchId` is required.
- `date` is required and must be `YYYY-MM-DD`.
- `categoryId` is optional.
- `lookbackDays` defaults to `30`.
- `lookbackDays` minimum is `7`.
- `lookbackDays` maximum is `180`.

The route is staff-authenticated and requires `analytics:read`.

## Response Shape

```json
{
  "branchId": "seed-branch-1",
  "forecastDate": "2026-05-05",
  "lookbackDays": 30,
  "expectedOrders": 82,
  "expectedRevenue": 1240.5,
  "items": [
    {
      "menuItemId": "burger-1",
      "name": "Classic Burger",
      "categoryName": "Main Courses",
      "expectedQuantity": 24,
      "expectedRevenue": 336,
      "confidence": "HIGH",
      "reason": "Based on average Tuesday sales over the last 30 days"
    }
  ],
  "hourlyDemand": [
    {
      "hour": 13,
      "expectedOrders": 18
    }
  ],
  "dataQualityWarnings": [
    {
      "code": "LOW_SAMPLE_SIZE",
      "severity": "MEDIUM",
      "message": "Only 2 matching weekday sales days were found, so confidence is limited."
    }
  ]
}
```

The shared TypeScript response type is `DemandForecastResponse`.

Each item `reason` is generated by the NestJS backend as a short manager-friendly sentence. It explains the quantity sold in the sample, the sample basis, and the resulting forecast quantity. Example: `20 sold over 3 similar Tuesdays. Forecast: 7.`

`dataQualityWarnings` explains forecast reliability limits such as low sample size, sparse history, no forecastable items, weather/event multiplier impact, or deterministic fallback-model usage.

## MVP Algorithm

For each request:

1. Validate the branch exists.
2. Derive `tenantId` from the branch.
3. Enforce staff tenant and branch access rules.
4. Parse the target `YYYY-MM-DD` date.
5. Determine the target weekday.
6. Build the lookback window ending before the target date.
7. Fetch only same-tenant, same-branch historical orders.
8. Include only valid sales orders:
   - order status `SERVED` or `COMPLETED`, or payment status `PAID`
   - exclude `CANCELLED`
   - exclude order payment status `REFUNDED`
9. Fetch order items linked to active menu items.
10. Apply optional category filtering.
11. Prefer same-weekday history when at least 3 matching weekday samples exist.
12. Fall back to all recent lookback days when same-weekday samples are too sparse.
13. Average item quantities across days with valid sales history in the selected sample.
14. Round expected quantities and expected orders to whole numbers.
15. Calculate expected revenue as `expectedQuantity * current menu item base price`.
16. Round expected revenue to 2 decimals.
17. Sort items by expected quantity descending.

For the MVP, sample days mean days with valid historical sales data. This keeps sparse demo and early-tenant forecasts visible while still keeping confidence low when same-weekday history is weak.

## Confidence Rules

Confidence is based on same-weekday sample availability:

```ts
if (sampleDays >= 6) confidence = "HIGH";
else if (sampleDays >= 3) confidence = "MEDIUM";
else confidence = "LOW";
```

When the service falls back to all recent days because same-weekday samples are sparse, item confidence remains low if fewer than 3 same-weekday sales-history days exist.

## Hourly Demand Logic

Hourly demand uses the same historical order set chosen for the forecast.

Timestamp priority:

1. `OrderStatusHistory.changedAt` for `COMPLETED`
2. `OrderStatusHistory.changedAt` for `SERVED`
3. completed `Payment.paymentDate`
4. `Order.orderDateTime`

Orders are grouped by UTC hour, averaged by sample days, rounded to whole orders, filtered to `expectedOrders > 0`, and sorted by hour ascending.

The frontend computes peak hour from `hourlyDemand`; it does not change backend forecast values.

## Tenant And Branch Safety

The branch is always loaded first, and its `tenantId` becomes the trusted scope for downstream queries.

Rules:

- Staff tenant must match branch tenant.
- Branch-bound staff can forecast only their own branch.
- `OWNER` and `MANAGER` can forecast other branches inside the same tenant.
- No query forecasts across tenants.
- All order and order-item reads include tenant and branch scope.

## Auditability

Forecast requests write `DemandForecastLog` records with:

- tenant ID
- branch ID
- requesting staff ID
- forecast date
- lookback days
- optional category ID
- expected orders
- expected revenue
- metadata about sample mode and historical order count

Audit logging is non-blocking. A log failure is warned but does not fail the forecast response.

Forecast audit metadata also records whether the deterministic fallback model was used and the aggregate data-quality warning codes returned to the UI.

## Frontend MVP

The admin analytics page renders `DemandForecastPanel`.

Controls:

- date picker, defaulting to tomorrow
- category selector from `/api/menu?branchId=...`
- lookback selector

Summary cards:

- Expected Orders
- Expected Revenue
- Peak Hour
- Top Item

Table columns:

- Item
- Category
- Expected Qty
- Expected Revenue
- Confidence
- Reason

The frontend calls only the NestJS API through the existing authenticated API client.

## Limitations

- Simple statistical average (with ML fallback support).
- Accuracy depends on order history quality.
- New items have low or no forecasted demand until enough sales history exists.
- Revenue uses current menu item base price only.
- Revenue does not include modifiers, tax, service charge, discounts, or refunds.
- Date and hour calculations dynamically use the `Branch.timezone` (falling back to UTC if missing) so accurate local days and peak hours are computed.
- Hugging Face is used exclusively for summarizing the narrative and never calculates raw numbers.
